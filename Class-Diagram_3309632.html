Travel Booking System : Class Diagram

Created by Anushka Shetty on Jun 09, 2023

Increment class diagram for Sprint 2

User and TravelPackage: This relationship is represented as an association. A user can browse, select, and interact with multiple travel packages, and a travel package can be associated with multiple users. It is a typical association where both classes can exist and function independently.

User and TravelPackage (Dependency): In the context of booking a travel package, we can also identify a weak relationship between User and TravelPackage. The User class may depend on the TravelPackage class for booking purposes. However, this dependency is not explicitly shown in the class diagram provided since it is not part of the current sprint.

TravelPackage and Flight: TravelPackage can include  Flights as part of its offering. The aggregation relationship signifies that the Travel Package is composed of Flights. Each Flight represents a part of the overall Travel Package. 

TravelPackage and Hotel: TravelPackage can include one or more Hotels as part of its offering. The aggregation relationship signifies that the Travel Package is composed of Hotels. Each Hotel represents a part of the overall Travel Package.

Travel Package and Activity: TravelPackage can include one or more Activities as part of its offering. The aggregation relationship signifies that the Travel Package is composed of Activities. Each Hotel represents a part of the overall Travel Package.

User and Booking: In the context of the travel booking system, the Booking and User classes have an association relationship rather than a dependency. The association represents the connection between the Booking and User classes, indicating that a Booking is associated with a specific User who made the booking.

Booking and Travel Package: The relationship is an association, which represents that a Booking is associated with a specific TravelPackage. This association indicates that a Booking is made for a particular TravelPackage.
Booking and EmailService: A dependency relationship indicates that one class depends on another class for some functionality or resources. In this case, the Booking class depends on the EmailService class to send email notifications to customers. The dependency relationship indicates that the Booking class uses the services provided by the EmailService class to send email notifications.
